# AiParser 聊天优先重构设计

## 1. 目标

将 `src/views/AiParser.vue` 从当前的“双栏工具页”重构为“聊天优先”的 AI 页面。

核心目标：

- 让 AI 对话流成为页面主内容区
- 流式回复采用接近 ChatGPT 的打字机式增量呈现
- 视频解析结果作为聊天流中的内联结果卡出现
- 下载动作直接放在结果卡中，不再放入右侧独立区域，也不使用弹窗
- `thinking/progress` 默认折叠，仅保留轻量状态提示

本次重构重点是信息架构和交互形态调整，不改变现有后端流式协议和下载接口语义。

## 2. 范围

### 2.1 包含

- 重构 `AiParser.vue` 页面布局为单列聊天流
- 引入最小必要的消息模型，承载用户消息与 AI 消息
- 将流式文本、过程状态、结构化结果写入当前 AI 消息
- 将解析结果以内联结果卡形式展示在聊天流中
- 将下载视频、下载音频按钮收敛到结果卡内部
- 优化移动端下聊天页和结果卡排版

### 2.2 不包含

- 不改造 `VideoParser.vue`
- 不引入 Pinia Store 管理 AI 会话
- 不接入完整的通用聊天系统能力
- 不新增多平台选择器
- 不新增下载弹窗、抽屉、右侧侧栏
- 不修改后端 SSE 协议字段

## 3. 设计结论

采用“单列聊天流 + 内联结果卡 + 结果卡内直接下载”方案。

选择原因：

- 最符合“AI 对话框优先”的产品目标
- 结果与上下文绑定，阅读链路自然
- 下载路径最短，避免多一次弹窗点击
- 相比右侧栏或底部抽屉，更像聊天产品而不是工具工作台

取舍：

- 结果卡会占用聊天流一定空间，但这是为了换取更强的上下文连续性
- 下载按钮直接暴露在卡片中，操作更快，但卡片字段必须严格收敛，避免视觉噪音

## 4. 页面结构

页面采用三段式结构：

### 4.1 顶部

- 保留 `Header`
- 页面标题区简化
- 标题文案突出“AI 解析”
- 辅助说明强调“输入链接或分享文案，由 AI 解析并直接返回可操作结果”

### 4.2 中部主区

单列聊天消息流，居中布局，宽度接近聊天产品阅读宽度，不再使用当前双栏布局。

消息流包含三类内容：

1. 用户消息
2. AI 文本消息
3. AI 消息中的内联结果卡

### 4.3 底部输入区

- 输入框固定在底部
- 支持多行输入
- 发送按钮固定在输入区右侧
- 发送中允许阅读历史消息，但禁用重复发送

## 5. 消息模型

页面内部新增最小必要消息模型，不上升为全局 Store。

```js
type MessageRole = 'user' | 'assistant'

type AssistantStatus = 'connecting' | 'streaming' | 'completed' | 'error'

interface ChatMessage {
  id: string
  role: MessageRole
  content: string
  createdAt: number
  status?: AssistantStatus
  progressMessage?: string
  thinkingText?: string
  thinkingExpanded?: boolean
  parsedData?: ParseResult | null
  toolStatus?: ToolStatus | null
  errorMessage?: string
}
```

说明：

- 用户消息只需要 `role + content`
- AI 消息统一承载文本、过程、结构化结果和错误状态
- 不单独拆出“结果卡消息”对象，避免结构过重

## 6. 流式交互设计

### 6.1 发送时

用户发送后，立即向消息流追加两条消息：

1. 一条用户消息
2. 一条空白 AI 消息容器

AI 消息容器初始状态为：

- `status = connecting`
- `content = ''`
- `progressMessage = '正在建立连接...'`

### 6.2 回复流式渲染

`reply_delta` 到达后，写入当前 AI 消息的 `content`。

展示策略：

- UI 呈现为打字机式输出
- 视觉上接近逐段落生成，而不是机械单字闪烁
- 如果底层已经是高频 delta，则前端只做平滑追加展示，不额外人为减速

### 6.3 过程信息

`progress` 与 `thinking_delta` 不再单独放在页面侧栏。

展示规则：

- 默认仅显示一条轻量状态文案，如“正在识别链接”或“正在准备结果”
- 在 AI 消息中提供“查看过程”折叠入口
- 展开后显示完整 `thinkingText` 与阶段性进度文本
- 默认折叠，避免破坏阅读主线

### 6.4 工具结果

收到 `tool_result` 且包含 `parsedData` 时：

- 将 `parsedData` 写入当前 AI 消息
- 将 `toolStatus` 写入当前 AI 消息
- 在当前 AI 消息正文下方渲染内联结果卡

关键原则：

- 结果卡属于当前 AI 消息的一部分
- 不跳转到独立区域
- 不脱离消息上下文单独展示

### 6.5 完成与失败

`done` 到达后：

- 合并最终文本与最终结构化结果
- 当前 AI 消息状态切为 `completed`

`error` 到达或请求异常时：

- 保留当前已收到的文本内容
- 当前 AI 消息状态切为 `error`
- 在该消息内显示失败提示
- 不清空整页，不丢失会话上下文

## 7. 结果卡设计

结果卡嵌入 AI 消息中，位于正文之后。

### 7.1 必须展示的字段

- 封面
- 标题
- 作者
- 时长
- 来源摘要
- 解析状态
- 风险提示
- 下载视频按钮
- 下载音频按钮

### 7.2 可收敛字段

以下字段不单独做大块展示，只在必要时以简短文案出现：

- `audioReady`
- `toolWarnings`
- `toolStatus.status`

例如：

- `audioReady === true` 时按钮文案显示 `下载原声音频`
- 否则显示 `提取并下载音频`
- `toolStatus.status === 'suspect'` 时，在卡片顶部展示一条警示条

### 7.3 风险与告警

如果存在以下条件，则在卡片顶部显示弱警告条：

- `toolStatus.status === 'suspect'`
- `toolWarnings.length > 0`

警告展示原则：

- 内聚在卡片顶部
- 一条区域承载，不拆多个大块
- 信息明确但不压过主内容

### 7.4 下载行为

下载按钮直接保留在结果卡中：

- `下载视频`
- `下载音频`

规则：

- 仅当 `parsedData.shareUrl` 存在时允许下载
- 若 `shareUrl` 缺失，则按钮禁用
- 禁用时在卡片内提示“当前解析结果暂不可下载，请重新解析或继续追问”
- 下载中只锁定当前按钮，不锁定整个页面

## 8. 状态与视觉层级

### 8.1 页面级状态

页面不再以“左边输入 / 左边结果 / 右边过程”分区承载状态，而是以消息为单位管理状态。

页面级只保留：

- `sessionId`
- 当前请求控制器
- 是否允许发送

### 8.2 消息级状态

每条 AI 消息单独持有：

- 文本内容
- 流式状态
- 过程信息
- 结果卡
- 错误信息

### 8.3 视觉层级顺序

1. AI 回复正文
2. 内联结果卡
3. 折叠的过程入口
4. 风险提示
5. 较弱的技术性状态标签

目标是让用户第一眼看见“AI 说了什么”，第二眼看到“现在能做什么”。

## 9. 移动端行为

移动端保持同一套聊天流结构，不额外切换为工具页。

适配规则：

- 结果卡改为上下结构
- 封面在上，文本在下
- 下载按钮改为纵向堆叠
- 输入区保持底部固定，但避免遮挡最后一条消息
- 折叠过程区域默认关闭

## 11. 验证标准

### 11.1 交互验证

- 页面主视图为聊天流，而非双栏工作台
- 用户发送后，聊天流中立即出现 AI 占位消息
- `reply_delta` 能持续驱动 AI 消息正文更新
- `parsedData` 返回后，结果卡出现在对应 AI 消息内部
- 下载按钮直接出现在结果卡中

### 11.2 状态验证

- 发送中不可重复发送
- 失败时保留已生成内容
- 重新发送不应破坏历史消息
- `thinking/progress` 默认折叠

### 11.3 边界验证

- `shareUrl` 缺失时按钮禁用
- `suspect` 状态有警示提示
- 下载中只影响当前按钮
- 移动端下结果卡按钮和输入区可正常操作

## 12. 原则对齐

### 12.1 KISS

- 只引入最小消息模型
- 不额外抽 Store
- 不新增弹窗、抽屉、侧栏等多套承载结构

### 12.2 YAGNI

- 不预留完整多轮聊天系统能力
- 不预留平台切换能力
- 不预留复杂消息插件体系

### 12.3 DRY

- 流式状态、工具结果、错误状态统一归属 AI 消息对象
- 下载逻辑继续复用现有 API 封装

### 12.4 SOLID

- 视图只负责消息渲染与交互编排
- 下载与流式请求继续依赖 API 抽象
- 单一消息对象承载单条 AI 回复的完整生命周期

## 13. 后续实现建议

2. 将流式状态迁移到消息模型
3. 实现内联结果卡与直接下载按钮
4. 收敛 `thinking/progress` 为默认折叠
5. 补齐移动端样式和回归验证
